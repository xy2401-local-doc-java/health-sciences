Registering a study

If you register an application whose directory structure conforms to the eNDA or eCTD standard, you can register studies automatically. If you do not register studies automatically, you must register them manually. You can manually register studies at any time after the application is registered.

When you register a study:

  • An Oracle account is created for the study.

The account name is derived from the study name that you provide during registration.

  • The study appears in the Select Study/Pool page.

1.         If the data source is SAS transport files, create and populate a study directory in the application directory.

A study directory is required for study metadata even if the data source is an Oracle database. For more information, see Directory Structure for Applications and Studies.

2.         Click the Setup tab.

The Application page appears.

3.         Click Studies/Pools below the Setup tab.

The Studies and Study Pools page appears.

4.         Click Register Study.

The Register Study page appears.

5.         Fill in the fields.

Note: Fields that are highlighted in yellow are required.

6.         If you are generating a new metadata file, click Generate Metadata.

After some time, the metadata file is generated, and the name of the new file appears in the Metadata field. The file name is in the following format: define_YYYYMMDDhhmmss.xml

A generated define.xml file uses the CDISC Operational Data Model (ODM) Version 1.2.1.

Note: If you specified SAS transport files as the Data Source Type, .xpt files must be present in the Data Location you specified. Otherwise, you encounter an error.

7.         To review the metadata file, hover over the View link, and click one of the following links:

  • Using standard stylesheet

  • Using sponsor-provided stylesheet

  • Raw XML

8.         Click OK.

Field Descriptions

Field

Description

Name

Name of the study.

Description

Description of the study.

SDTM version

Version of CDISC SDTM associated with the study. For example, sdm312 indicates version 3.1.2.

The default is the SDTM version that was specified for the application.

If a study is loaded with an existing define.xml and the file references an SDTM version, the study is associated with that SDTM version. The value you specify in this field is overridden.

MedDRA account

Name of the Oracle account that contains the MedDRA dictionary used by the study. The default is the MedDRA Account that was specified for the application.

If a study is loaded with an existing define.xml file and the External Codelist element for the AEDECOD variable references a MedDRA version (account) that exists, Empirica Study assigns that MedDRA version to the study. The value you specify in this field is overridden.

Note: If your organization has supplied a define.xml file, you might want to view it before registering the study to determine the MedDRA version (if any) specified by the define.xml so that you can set this field to match. See the description of the Metadata field below for instructions on viewing the define.xml file.

Indication

Indication, if any, associated with the study.

Check variable labels against SDTM
  • If selected—A rule checks whether the variable descriptions in the define.xml file match the standard variable descriptions according to the SDTM version for the study. If any descriptions do not match, a checking result is reported on the Study Data Domains page of the Domains tab.
  • If deselected—The variable descriptions are not checked.

Reference ID Options

When you click Show Details, the following options appear:

  • Reference ID variable:
    • REFID—The value of __REFID variable from the study data appears by default on the Subject Details page for each relevant domain.
    • SPID—The value of __SPID variable from the study data appears by default on the Subject Details page for each relevant domain.

If the Reference ID variable — by itself, or when prefixed by the root for URLS — contains a valid Internet address, the Subject Details page displays a hyperlink that resolves to that Internet address.

  • Column header for 2nd level drilldown—Specify the column heading for the reference ID variable as you want it to appear on the Subject Details page (that is, second-level drilldown). If you leave this field blank, __REFID or __SPID will be used as the column heading.

  • Root for URLs—Root URL to be added as a prefix to all values of the __REFID or __SPID variable from the study data. This value is needed only if the study data contains only partial URLs for an external system.

Note: This option affects only the Subject Details page. It does not affect the Clinical Data page.

  • Display content as—One of the following, indicating what will appear in the reference ID variable column (depending on your selection) on the Subject Details page if there is a __REFID or __SPID value in the study data:

    • Value in REFID/SPID column—The value of __REFID or __SPID from the study data.
    • Image file—The graphic from the specified file. Specify only the file name. The file must be located in the image subdirectory of the web_root directory that was established for the instance during installation. For standard installations, this is /u01/stage/Study_Install/Study/image.
    • Fixed label—The specified text string.

Data source type

Available only if the Allow data import from Oracle Database site option is selected.  The Data Source Details required to register a study vary depending on the data source type selected:

  • SAS Transport Files—The study data is in .xpt files that are located in the specified data location.
  • Oracle Database—The study data is from an Oracle database, and is in an accessible Oracle database account.

When using Oracle views for Oracle Database data, the efficiency of the views has a major impact on how long it will take to generate the define.xml.

Note: All data sources, regardless of type, must be formatted for SDTM compliance.

Data location

Name of the study directory relative to the application directory. Click Browse to navigate to the application directory and select a directory. For more information, see Directory Structure for Applications and Studies.

If there are define.xml files in this location, you can select them in the Metadata field. If you generate a new define.xml file, the file is generated in this location.

If the study’s data source type is SAS transport files, you must place the *.xpt files in this location before loading the study.  For information on loading a domain's data from multiple xpt files, see Split Domains.

For SAS transport files—Specifies the directory where both study data files and the study metadata (define.xml) reside.

For an Oracle database—Specifies the directory where metadata (define.xml) for the study resides.

Metadata

Specifies metadata generation method, or an existing metadata file. The options are:

  • Generate a new metadata file with standard codelists—Create a new define.xml file that includes a standard CDISC compliant codelist for each variable that is subject to CDISC controlled terminology. For more information, see Codelists.
  • Generate a new metadata file with standard and data-driven codelists—Create a new define.xml that includes:
  • A standard, CDISC-compliant codelist for each variable subject to CDISC controlled terminology.
  • A codelist containing all unique non-null values from the study data for that variable for each variable subject to sponsor-defined controlled terminology. For more information, see Codelists.
  • Use Define.xml—Available if the study directory (indicated in the Data location field) contains a file named define.xml. There may be multiple define.xml files. When a define.xml file is generated, the file name includes the date and a numeric identifier.
  • If you provide your own define.xml file, it can use ODM Version 1.2.1 or 1.3.

Note: If there is only one define.xml in the study directory, that file is the default in this field. If there are no define.xml files in the study directory, the default in this field is determined by the application's setting for Default codelists in metadata.

When a define.xml file name appears in this field, you can hover your mouse over the View link to view the define.xml file in one of the following formats:

  • Using standard stylesheet—Always available. Presents the metadata using a standard built-in style sheet.
  • Using sponsor-provided stylesheet—Available only if you loaded the study with a define.xml file that your organization provided (that is, it was not generated by Empirica Study), and the define.xml referenced a style sheet. The style sheet reference can be to a file location in the study data directory or a subdirectory in the study data directory.
  • RAW XML—Always available. Presents the metadata without using a style sheet.

References to SAS transport files and an annotated CRF file (if any) must be to the study directory or to a subdirectory within the study directory.

Note: To view the annotated CRF, you must have Adobe Acrobat Reader installed. To view SAS transport files, you must have the SAS Universal Viewer or Base SAS installed. For more information, see Prerequisites and Usage Notes.

Database

Reference to the Oracle database in which the study resides. If the study is hosted in the same database as Empirica Study, select <local>. Alternatively, choose the name of the database link that is defined on the Empirica Study database and points to the Oracle database.

Account

Name of the Oracle account that owns the Oracle Database data structures.

You can click Select to select an account from a list of accounts containing a table or view named CV_SDTM<SDTM version>_DM, where <SDTM version> is 311, 312, 313, or 32 (depending on the SDTM version specified for the study).

Note: If none of the accounts contain views with names that start with the SDTM version for the study, the list is empty.
