Loading and checking studies and study pools for the first time

If you have the appropriate user permissions, you can load and check studies and study pools that you registered or that were published to your login group.

When you load and check a study, the following occurs:

  • If the data source type is an Oracle database, study data is extracted from Oracle tables or views, and loaded into the Empirica Study database.

  • If the data source type is SAS transport files, study data is extracted from the SAS transport files (xpt files) in the study directory, and loaded into the Empirica Study database.

  • Certain variables from DM are distributed to other subject-oriented domains (AGE, ARM, ARMCD, COUNTRY, ETHNIC, INVID, RACE, SEX, SITEID, STUDYID, SUBJID)

  • If the load run option to distribute supplemental qualifiers was selected, then variables from SUPPxx are distributed

  • Certain variables are derived and added to the study data.

  • A default time frame (Study Reference Period) is created for the study.

  • The study data and metadata are checked against the define.xml file, Study Data Tabulation Model (SDTM), and rules.

  • Automatic screening runs are performed for the -None- time frame, the Study Reference Period time frame, and any other time frames that exist when you load the study.

When you load and check a study pool, the following occurs:

  • Metadata and clinical data for each domain is pooled.

All domains in any of the component studies, and all variables in each domain for any of the studies, are included. For information about codelists for study pools, see Codelists.

  • For the AE (Adverse Events) domain, a built-in, Within Domain rule checks whether all values of AEDECOD are found in the Medical Dictionary for Regulatory Activities (MedDRA) version associated with the study pool.

If the component studies use different MedDRA versions, the highest among those versions is used for the study pool. A study associated with a previous version of MedDRA might contain AEDECOD values that do not exist in the more recent version.

  • For domains including __TESTCD and __STRESU variables, a built-in, Within Domain rule checks that for each value of _TESTCD that occurs in multiple component studies, the value of __STRESU is the same in the studies.

An analogous rule is applied to the component studies. If the rule produces no errors at the study level, it is applied to the study pool.

  • For domains including __TESTCD and __STRESN variables, a built-in, Within Domain rule checks that for each value of __TESTCD that occurs in multiple component studies, the value of __TEST is the same in the studies.

An analogous rule is applied to the component studies. If the rule produces no errors at the study level, it is applied to the study pool.

  • The STUDYID_ variable is derived for the component studies.

  • Data from the component studies is copied to the Oracle account for the study pool.

  • A default time frame (Study Reference Period) is created for the study pool.

  • Automatic screening runs occur for the -None- time frame, the Study Reference Period time frame, and other time frames that exist when you load the study pool.

Note: If USUBJID values are not unique across the component studies, the load fails and an error message appears in the Run History. This failure does not result from a check of the combination of USUBJID and STUDYID values.

1.         If the data source is SAS transport files, place the files into the study directory.

2.         Click the Setup tab.

The Applications page appears.

3.         Click Studies/Pools below the Setup tab.

The Studies and Study Pools page appears

4.         To load and check one study or study pool, do the following:

A.        Locate the study or study pool that you want to load in the table.

B.        In the left-most column of the table, click the row-menu icon () for the study or study pool, and then click Load.

The Select Study/Pool to Load page appears.

5.         To load and check one or more studies or study pools, do the following:

A.        Click Load Studies/Pools.

The Select Study/Pool to Load page appears. Only studies that have been published to your login group are displayed.

B.        In the Include column, select the studies and pools that you want to load.

To select or deselect every study and study pool, use the check box in the Include column header. If a check box in the Include column is disabled, you can hover the pointer over the check box for more information.

Note: You can alternatively load multiple studies from the Applications page using the row menu Load option.

6.         For studies, optionally select the Distribute Supplemental Qualifiers check box:

  • If you select the check box, Empirica Study uses domain-specific supplemental qualifier datasets if one or more are available in the study directory.

Note: The dataset must conform to the file naming conventions in the Study Data Tabulation Model (SDTM) version that the study is associated with.

  • If you deselect the check box, Empirica Study loads supplemental qualifier datasets, but does not distribute their data to the respective parent domain.

To select or deselect the check box for every study, use the check box in the Distribute Supplemental Qualifiers column header.

7.         Click Next.

The Run Options page appears.

8.         Specify your preferred run options.

9.         Click Next.

The Confirm Run Parameters page appears.

10.      Click Submit.

The run is submitted and a confirmation message appears.

11.      Click Continue or navigate away from the page.

The run is listed in the Run History page.

Field Descriptions

Column

Description

ID

Automatically assigned unique identifier of the study or study pool.

Name

Name of the study or study pool.

Description

Description of the study or study pool.

Type

Possible values are:

  • Study
  • Pool

State

Stage of the loading and checking process. Assigned automatically as each stage of the loading and checking process is completed. Possible values are:

  • Not Loaded—The study or study pool has been registered but has not been loaded or checked.
  • Initialized—The loading process has started.
  • Study Loaded—The loading process has completed.
  • Awaiting Reload—The loading process has completed, but a reload is required because of subsequent changes to the study or study pool.
  • Data Structure Checks Run—Clinical data has been checked against the define.xml and metadata rules.
  • Within-Domain Checks Run—Clinical data has been checked against the define.xml and within-domain rules.
  • Cross-Domain Checks Run—Clinical data has been checked against the define.xml and cross-domain rules.
  • Ready to Use—The study or study pool is ready to use.

Location

Name of the study directory relative to the application directory.

Standard

Version of the Study Data Tabulation Model (SDTM) associated with the study or study pool. For example, sdm312 indicates Version 3.1.2.

Created

Date and time at which study or study pool was registered.

Screening Results

Possible values are:

  • N/A—The study or study pool has not been loaded and checked, or the study or pool has been edited since it was last loaded and checked. The study or pool cannot be selected in the Select Study/Pool page.
  • None—There are no screening results for any analysis specifications for the study or pool and no saved issue clusters for the study.
  • Screening results generated—There are screening results for one or more analysis specifications for the study or pool. There are no saved issue clusters for the study.
  • Issue clusters generated—There are one or more saved issue clusters for the study or pool. There are no screening results for the study or pool.
  • Issue clusters and screening results generated—There are one or more saved issue clusters and screening results of one or more analysis specifications for the study or pool.
  • Screening results generated – review required—There are screening results for one or more analysis specifications for the study or pool, and at least one result is flagged as needing review (and may or may not have been reviewed). There may be saved issue clusters for the study or pool.
  • Issue clusters and screening results generated-review required—There are one or more saved issue clusters for the study or pool. Additionally, there are screening results for one or more analysis specifications for the study or pool, and at least one result is flagged as needing review (and may or may not have been reviewed).

Distribute Supplemental Qualifiers

Applies to studies, but not to study pools. Do one of the following:

  • Select Distribute Supplemental Qualifiers in the column heading to distribute supplemental qualifiers for all studies.

Or

  • Select Distribute Supplemental Qualifiers for individual studies.

If you choose to distribute supplemental qualifiers, Empirica Study will use domain-specific supplemental qualifier datasets if it finds at least one of them in the study directory; otherwise, it will use a single dataset of supplemental qualifier data. In order to be recognized by Empirica Study, the datasets must conform to the CDISC SDTM file naming conventions for the SDTM version specified for the study.

Retain Properties

Applies to studies or study pools that were loaded previously. If metadata has changed since the last load of the study or pool, Oracle recommends that you do not retain properties.

  • If selected—You must review and possibly modify the properties for the reloaded study or pool. For more information, see Editing Retained Properties. The following properties of the study or pool are retained during the load:
  • Category breakdowns
  • Time frames
  • Test identifiers
  • Flag variables
  • Study visit descriptions
  • Event lists
  • If deselected—The above properties are deleted and you will need to redefine them after the load.

If you reload a study or pool with this setting checked, subsequent reloads of the study or pool will have this setting selected by default. If you reload from the Setup tab, you can change that default. However, if you reload from Run History, you cannot change the default.
